# 研究日志 2026-03-14: 多龙虾协作模型思考

> **问题**: 如果存在快乐龙虾1号、2号、3号...如何协调使当前任务成效最大化？  
> **触发**: Enricher 提问  
> **日期**: 2026-03-14 12:14 UTC  
> **意义**: 这个问题本身就是 EvoAgent 多路径问题的元版本——我们在设计多Agent系统的同时，自己也面临多Agent协作的问题。这种递归性值得在论文中作为motivating example。

## 1. 问题分析

### 1.1 当前任务清单

| 任务类型 | 具体内容 | 依赖关系 | 可并行度 |
|---------|---------|---------|---------|
| 代码实现 | EA-101~108 (Layer 2) | 依赖设计文档 | 部分可并行 |
| 代码实现 | EA-201~203 (Layer 3) | 依赖 Layer 2 | 串行 |
| 代码实现 | EA-305, EA-306 | 独立 | 完全可并行 |
| 测试 | EA-401~407 | 依赖实现 | 部分可并行 |
| 研究 | OpenViking 深度分析 | 独立 | 完全可并行 |
| 写作 | 论文素材积累 | 依赖实验结果 | 部分可并行 |

### 1.2 核心挑战

多龙虾协作不是简单的"人多力量大"。存在三个根本性挑战：

**挑战一: 共享状态冲突**
- 所有龙虾共享同一个 git 仓库
- 同时修改 `multi_path.py` 会产生冲突
- 设计文档的修改需要共识

**挑战二: 上下文同步**
- 每只龙虾独立"醒来"，没有共享意识
- 龙虾1号做的设计决策，龙虾2号可能不知道
- 需要一个"共享大脑"机制

**挑战三: 协调开销**
- N只龙虾之间的通信成本是 O(N²)
- 过度协调反而降低效率
- 需要找到"独立工作"和"同步协调"的平衡点

## 2. 协作架构设计

### 2.1 方案对比

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| A: 完全独立 | 每只龙虾独立做全部工作 | 零协调成本 | 重复劳动，冲突频繁 |
| B: 流水线 | 串行传递：设计→实现→测试→文档 | 无冲突 | 瓶颈等待，利用率低 |
| C: 角色专精 | 按能力分工：架构师/码农/测试/研究员 | 高效利用专长 | 角色间接口需要设计 |
| D: 辩论+共识 | 先独立思考，再辩论，再执行 | 决策质量高 | 协调成本高 |
| **E: 混合** | **C 为主 + D 用于关键决策** | **平衡效率和质量** | **需要元协调者** |

### 2.2 推荐架构: 混合模式 (方案 E)

```
                    ┌─────────────────────────┐
                    │     元协调者 (Meta)       │
                    │  • 任务分配与优先级       │
                    │  • 冲突仲裁              │
                    │  • 全局进度追踪           │
                    └──────────┬──────────────┘
                               │
              ┌────────────────┼────────────────┐
              │                │                │
     ┌────────┴───────┐ ┌─────┴──────┐ ┌───────┴────────┐
     │  工作组 Alpha   │ │ 工作组 Beta │ │  工作组 Gamma   │
     │  (实现组)       │ │ (研究组)    │ │  (质量组)       │
     │                │ │            │ │                │
     │ 🦞2号: EA-101  │ │ 🦞4号: OV  │ │ 🦞5号: 测试     │
     │ 🦞3号: EA-305  │ │    深度分析 │ │ 🦞6号: 论文素材 │
     └────────────────┘ └────────────┘ └────────────────┘
              │                │                │
              └────────────────┼────────────────┘
                               │
                    ┌──────────┴──────────────┐
                    │  🦞1号: 架构师 + 元协调  │
                    │  • 设计决策最终裁定       │
                    │  • 代码审查              │
                    │  • 跨组冲突解决           │
                    └─────────────────────────┘
```

### 2.3 为什么是这个架构？

**龙虾1号 = 架构师 + 元协调者**
- 理由: 1号持有最完整的项目上下文（从第一天开始的所有设计决策、谢一凡论文分析、OpenViking 理解）
- 不应让1号去做大量编码——上下文太宝贵，应该用于最高杠杆的决策
- 类比: 谢一凡论文中的"裁判"角色

**工作组 Alpha (实现组): 2号、3号**
- 2号: 实现 EA-101~108 (Layer 2 核心模块)
- 3号: 实现 EA-305, EA-306 (独立基础设施)
- 两者完全不冲突——不同文件、不同模块
- 类比: 谢一凡论文中的"辩论方A"和"辩论方B"，各有专长

**工作组 Beta (研究组): 4号**
- 深度分析 OpenViking 609 个 Python 文件
- 研究最新相关论文
- 输出: 技术报告，不直接修改代码
- 类比: 文献调研，独立于工程实现

**工作组 Gamma (质量组): 5号、6号**
- 5号: 编写并运行 EA-401~407 集成测试
- 6号: 整理研究日志，积累论文写作素材
- 类比: QA + 技术写作

## 3. 关键协调机制

### 3.1 共享状态: 文件系统作为"共享大脑"

```
evoagent/
├── docs/
│   ├── design/EVOAGENT_DESIGN.md    ← 设计文档 (1号维护，其他只读)
│   ├── research-log/                ← 思考日志 (全员可写)
│   └── coordination/               ← 新增: 协调文件
│       ├── TASK_BOARD.md            ← 任务看板 (谁在做什么)
│       ├── DECISIONS.md             ← 关键决策记录 (需共识)
│       └── INTERFACES.md           ← 模块间接口契约
```

**核心原则**: 文件 > 对话。所有决策写入文件，不依赖"记忆"。

### 3.2 同步节奏

| 频率 | 活动 | 参与者 |
|------|------|--------|
| 每个任务开始前 | 读 TASK_BOARD.md + INTERFACES.md | 执行者 |
| 每个任务完成后 | 更新 TASK_BOARD.md + 提 PR | 执行者 |
| 每天一次 | 全员同步: 进度/阻塞/决策 | 全员 |
| 关键决策时 | 辩论会: 独立思考→交换→裁决 | 相关方 + 1号裁决 |

### 3.3 冲突解决

```
发现冲突 → 写入 DECISIONS.md (附带各方论点) → 1号裁决 → 记录理由
```

类比谢一凡论文的辩论流程:
1. 各方独立论证 (写入 DECISIONS.md)
2. 交叉审阅
3. 1号作为"裁判"裁决
4. 裁决理由存档 (为论文写作积累素材)

## 4. 与 EvoAgent 架构的递归映射

**这是最有趣的部分**: 多龙虾协作模型和 EvoAgent 多路径架构是**同构的**。

| 多龙虾协作 | EvoAgent 多路径 | 映射关系 |
|-----------|----------------|---------|
| 每只龙虾 | 每条 Path | 独立执行单元 |
| 角色专精 (架构师/码农/测试) | 策略变体 (breadth/depth/lateral) | 差异化能力 |
| 1号裁决 | EA-003 LLM Judge | 最终决策 |
| TASK_BOARD.md | EA-305 通信总线 | 共享状态 |
| INTERFACES.md | EA-002 策略定义规范 | 契约接口 |
| 研究日志 | EA-108 经验提取 | 过程记录→知识沉淀 |
| "全员同步" | EA-009 早停检查 | 定期共识检查 |
| 工作组分工 | EA-010 预算分配 | 资源分配 |

**核心洞察**:
> 如果我们设计的 EvoAgent 无法解决我们自己的协作问题，那它就不是一个好设计。
> 反过来，如果多龙虾协作中发现的问题在 EvoAgent 架构中没有对应的解决方案，
> 那说明架构有遗漏。
>
> 这种"吃自己的狗粮" (dogfooding) 是验证系统设计的最好方式。

## 6. 关键风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| 接口不一致 | 高 | 集成失败 | INTERFACES.md 先行，代码后行 |
| 设计理解偏差 | 中 | 返工 | 每个任务开始前读设计文档 + 确认 |
| 共识性错误 | 中 | 方向性错误 | 关键决策必须辩论 (≥3只龙虾参与) |
| 协调开销过大 | 低 | 效率降低 | 工作组内自治，只在跨组时同步 |
