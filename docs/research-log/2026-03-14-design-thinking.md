## 2. 关键思考过程

### 2.1 OpenViking 的启发：文件系统范式统一存储

**问题**: EvoAgent 的进化数据（战绩、画像、经验）应该存储在哪里？

**思考过程**:
- 最初考虑 JSONL 文件或 SQLite 数据库
- 分析 OpenViking 后发现其"文件系统范式" (`viking://` URI) 可以统一管理 Memory/Resource/Skill
- 关键洞察：**L0/L1/L2 分层加载天然适合进化数据的成本控制**
  - 日常策略选择只需读 L0 (~100 tokens)："breadth_first 胜率 72%"
  - 深度分析才需要 L2：完整执行日志
  - 这比"要么全加载要么不加载"的方案节省 50-70% Token

**设计决策 DD-006**: EA-307 (OpenViking) 作为进化系统的唯一存储层。

**思考日志**:
> 一开始觉得 OpenViking 只是"另一个 RAG 框架"。但仔细分析后发现它的核心创新不是检索算法，
> 而是**组织范式**——用目录结构+分层摘要来管理 Agent 的全部上下文。
> 这正好解决了 EvoAgent 的数据碎片化问题：
> 战绩在一个 JSON 里，画像在另一个 JSON 里，经验在 Markdown 里...
> 统一到 viking:// 目录结构后，一个 `load_task_context()` 调用就能拿到所需层级的全部信息。

### 2.2 Self-Improving Agent 的启发：经验进化 + 竞争进化

**问题**: EvoAgent 的进化应该只基于"谁赢了"吗？

**思考过程**:
- 最初设计只有竞争进化：记录投票结果，统计胜率，选择最优策略
- 分析 Self-Improving Agent 后发现其核心理念是 **从错误中学习**
- 关键洞察：**"谁赢了"和"为什么赢了"是两个不同维度的进化信号**

**融合两种范式的思考**:

```
竞争进化 (原有)        经验进化 (Self-Improving 借鉴)
"谁赢了？"            "为什么赢了？"
  投票结果               执行日志分析
  统计胜率               提取失败模式
  ↓                     ↓
  选择最优               避免重复错误
```

> **关键区别**: Self-Improving Agent 是**被动式**——等错误发生再记录。
> EvoAgent 应该是**主动式**——主动尝试多种策略，对比效果，自动进化。
> 
> 这是本质区别：**Self-Improving = 从失败中学习**，**EvoAgent = 从竞争中进化**。
> 
> 最优方案是融合两者：
> - 竞争进化提供"谁赢了"的信号
> - 经验进化提供"为什么赢了"的深层分析
> - 两者共同驱动策略画像的更新

**设计决策 DD-007**: 融合竞争进化 + 经验进化双范式。

### 2.3 Self-Improving Agent 的三层晋升 → EvoAgent 的策略生命周期

**思考过程**:

Self-Improving 的晋升路径：
```
L1 记录 (.learnings/) → L2 提升 (SOUL/AGENTS/TOOLS) → L3 技能提取 (独立 Skill)
```

映射到 EvoAgent：
```
L1 战绩记录 (strategy_results/) → L2 策略画像 (strategy_profiles/) → L3 新策略生成 (evolved_xxx/)
```

> 这个映射非常自然：
> - Self-Improving 的 Learning Entry 格式 (ID/Priority/Status) 直接可用于 EA-108 经验记录
> - `Recurrence-Count` 字段完美解决了"何时触发新策略生成"的问题——同类失败 ≥ 3 次
> - `Pattern-Key` 字段可用于策略淘汰的依据——标记反复出现的失败模式

### 2.4 EA-307.5 自迭代作为进化心跳

**问题**: 进化系统需要一个独立的调度器吗？

**思考过程**:
- 方案 A: 新建一个 cron job 定期触发进化
- 方案 B: 利用现有 EA-307.5 `trigger_memory_iteration()` 
- 选择 B：无需额外调度器，利用 OpenViking 已有的自迭代机制

> 关键思考：进化不应该是一个独立的系统，而是**存储层的自然行为**。
> OpenViking 的 memory self-iteration 设计初衷就是"会话结束后自动提取长期记忆"。
> 对 EvoAgent 来说，"会话结束后自动提取策略经验"是同一个逻辑。
> 因此 EA-307.5 天然就是进化系统的心跳。

**设计决策 DD-008**: EA-307.5 自迭代作为进化系统心跳。

### 2.5 EA-104 自适应策略选择：Exploit + Explore 平衡

**问题**: 如何从静态策略轮转变为动态策略选择？

**思考过程**:
- 当前实现是 `STRATEGY_VARIANTS[:num_paths]`，即固定取前 N 个
- 这无法利用历史数据
- 类比 Multi-Armed Bandit 问题：每个策略是一个"臂"，胜率是回报

> Exploit (利用): 选择历史最优策略 → 短期最优
> Explore (探索): 尝试低频/新策略 → 长期最优
> 
> 纯 Exploit → 陷入局部最优，永远不会发现更好的策略
> 纯 Explore → 浪费资源在已知劣势策略上
> 
> 平衡方案：
> - 路径 1-2: Exploit (历史最优)
> - 路径 3+: Explore (低样本策略 or 新进化策略)

**设计决策 DD-009**: 策略选择采用 Exploit + Explore 平衡。

### 2.6 谢一凡论文的启发：提问框架 → 经验提取结构

**问题**: EA-108 经验提取时，如何结构化地分析"为什么失败"？

**思考过程**:
- 论文的 CoT 五步模板提供了一个现成的结构化分析框架
- 论文发现 CoT 效果与任务复杂度正相关 → 验证了 EA-104 自适应选择的核心假设
- 论文的三类错误 (共识性/裁判失误/论证不均) → 可直接作为 EA-108 的 Pattern-Key

**映射关系**:

| 论文框架 | EvoAgent 应用 |
|---------|--------------|
| CoT 五步模板 | EA-108 经验提取的分析结构 |
| 辩论五阶段 | EA-307.5 自迭代的评估流程 |
| 三类错误分类 | EA-108 Learning Entry 的 Pattern-Key |
| "任务复杂度决定方法" | EA-104 自适应选择的理论支撑 |

**关键发现**:

> 论文中辩论系统的三个错误类型对 EvoAgent 有重要警示：
> 
> 1. **共识性错误** — 当所有 path 都用相同策略变体时，可能都犯同样的错。
>    → EvoAgent 的 Explore 机制正是为了避免这种"群体思维"
> 
> 2. **裁判判断失误** — LLM Judge 可能被"专业包装"的错误答案误导。
>    → EA-003 的 Judge prompt 需要强调"实质准确性 > 表述自信度"
> 
> 3. **论证质量不均** — "伪精确"的错误论证 > 全面但模糊的正确论证。
>    → 投票机制需要多维评分，不能只看"谁说得更自信"
